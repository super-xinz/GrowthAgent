# ThreadPilot 增长工作流重设计

## 目标

让用户无需命令行即可从任一产品启动小红书机会搜索；高意向机会自动生成可审核草稿；人工确认后能够可靠执行一次评论或回复。

## 信息架构

Dashboard 以增长工作流而非系统指标为主：产品卡片显示高意向机会、待审核草稿和已发布数量，并提供“开始寻找机会”主按钮。产品页顶部提供关键词、详情读取数量与 70 分自动草稿阈值。机会页使用“全部、高意向、等待确认、已发布”视图。

## 搜索与自动草稿

用户从 Dashboard 或产品页输入关键词并启动搜索。后端导入笔记与评论、去重、评分；机会分数不低于 0.70 时立即通过已配置 LLM 生成草稿并持久化。低于阈值的机会仅保存内容，允许稍后手动生成。

草稿必须先提供独立价值，再在目标需求与 Product Brain 已验证能力匹配时自然提及产品。只有产品明确标记为我方产品时，草稿才允许使用用户配置的真实关系说明；未验证所有权时不得冒充官方、开发者或团队成员。

## 人工确认与发布

高意向机会直接展示可编辑草稿和“审核并发布”。任何编辑都会使旧确认失效。确认令牌绑定机会、草稿哈希、当前小红书账号，十分钟有效且只能使用一次。

执行前进行只读预检：用机会来源关键词重新搜索目标笔记，按笔记 ID 获取最新 `xsecToken`，读取详情并确认目标评论仍存在。随后只调用一次评论或回复写接口。写操作不重试。成功后记录发布时间和状态；失败返回可操作的中文错误。

## 产品身份

产品增加 `is_owned` 布尔值，默认 false。关系说明默认空。只有 `is_owned=true` 且关系说明非空时，草稿可以进行产品引导；否则只提供中立帮助，不进行虚假推广。

## 错误处理

- 未登录：引导账号页扫码；
- 搜索无结果：保留关键词并给出换词建议；
- 旧 token：发布预检刷新，不直接使用入库 token；
- 目标评论已删除：阻止发布并标记目标失效；
- 草稿不合格：最多重新生成一次，仍失败则显示原因；
- 达到每日上限或全局停止：发布前拒绝。

## 验证

后端覆盖 token 刷新、评论存在性、单次写调用、自动草稿阈值、去重、身份边界、账号绑定和每日上限。前端覆盖工作流汇总、筛选逻辑、搜索表单状态和自动草稿展示。最终运行完整 Pytest、Ruff、前端测试、ESLint、TypeScript、npm audit、Docker 生产构建和本地健康检查。
