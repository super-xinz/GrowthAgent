# 产品排序、软删除与七天保留设计

## 目标

产品可以在总览中通过拖拽或上移、下移调整顺序，切换器与总览始终使用同一持久化顺序。删除产品时先进入回收站，保留七天；期间可以恢复或立即永久删除，过期后由后台任务自动永久清理。

## 数据模型

`products` 增加三个字段：`sort_order` 为非空整数，决定正常列表和回收站列表的显示顺序；`deleted_at` 为可空时区时间，非空代表软删除；`purge_after` 为可空时区时间，软删除时固定设置为 `deleted_at + 7 天`。

创建产品时，`sort_order` 设置为当前未删除产品最大顺序加一。现有产品迁移时按创建时间建立稳定顺序。恢复产品时清空删除时间，并将其放到正常列表末尾，避免与现有顺序冲突。

## API

- `GET /v1/products`：只返回未删除产品，按 `sort_order`、`created_at` 排序。
- `GET /v1/products/trash`：只返回已删除且未永久清理的产品，按 `deleted_at` 倒序。
- `PUT /v1/products/order`：接收完整的未删除产品 ID 数组。必须与数据库中的未删除产品集合完全一致；后端在一个事务中写入连续顺序。
- `DELETE /v1/products/{id}`：软删除，设置 `deleted_at` 与七天后的 `purge_after`，并暂停产品运行和自动发布。
- `POST /v1/products/{id}/restore`：恢复产品，清空删除时间并追加到正常列表末尾。
- `DELETE /v1/products/{id}/permanent`：只允许永久删除已经位于回收站的产品；数据库级联清理关联数据。

普通产品详情和所有产品业务接口均把已软删除产品视为不存在。排序请求失败时不允许部分更新。

## 七天自动清理

Celery Beat 每小时触发一次清理任务，删除所有 `purge_after <= 当前时间` 的产品。任务必须幂等：重复运行不会报错或影响正常产品。API 启动时保持现有表结构兼容逻辑，Alembic 迁移作为正式数据库升级路径。

## 前端交互

Dashboard 产品卡片增加拖拽手柄、上移、下移和删除操作。拖拽使用原生 Pointer/Drag 事件，不引入新的 UI 依赖；上移与下移按钮提供移动端和键盘等价操作。排序先在界面中乐观更新，API 失败时恢复原顺序并显示错误。

删除使用确认对话框，明确写出产品名称、将立即从工作区隐藏、七天后永久删除。成功后刷新根布局数据；若从当前产品页面删除，跳转到 `/dashboard`。

Dashboard 增加“回收站”入口与回收站面板。每条记录显示删除时间和剩余天数，提供“恢复”和“立即永久删除”。永久删除需要二次确认并明确不可恢复。

产品切换器只显示未删除产品，并遵循后端顺序。切换器不提供排序或删除操作，以避免拥挤；这些管理操作集中在 Dashboard。

## 错误与并发

排序 API 使用完整 ID 集合校验，防止两个页面同时排序造成遗漏。若另一个会话新增、删除或恢复了产品，旧页面的排序请求返回 `409`，前端刷新产品列表并提示用户重试。

重复软删除、恢复不存在的产品或永久删除正常产品分别返回明确的 `404` 或 `409`。清理任务执行永久删除时依赖数据库外键级联，不留下孤立数据。

- 前端纯函数测试覆盖上移、下移、拖拽重排和剩余天数显示。
- 浏览器回归覆盖 Dashboard 排序、刷新后顺序保持、软删除、回收站恢复、永久删除确认、切换器同步及窄屏按钮。
- Ruff、Pytest、前端测试、ESLint、TypeScript 和 Next.js 生产构建必须全部通过。
- Reddit 自动发布与审批安全门禁保持不变。
